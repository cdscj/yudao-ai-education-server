## 1. 先搞懂：AI 聊天到底是怎么回事？

### 1.1 用生活例子理解

想象你去一家餐厅吃饭：

```
你："老板，来一碗牛肉面"
老板：（去厨房做面...）"好嘞，您的牛肉面！"
```

这就是一次完整的AI对话。但是注意——**老板做完你的面之后，他就忘了你是谁**。你下次再来，他不会记得你上次点了什么。

AI 也是这样的！每次你发消息给 AI，它就像一个"失忆的厨师"：
- 你发什么，它回答什么
- 回答完就忘了
- 下次对话从零开始

### 1.2 那 AI 怎么能"记住"之前的对话呢？

很简单：**每次聊天时，把之前的聊天记录一起发给 AI**。

```
你第1次: "我叫小明"
AI: "你好小明！"

你第2次实际发给AI的内容:
  [历史记录]
  用户: "我叫小明"
  AI: "你好小明！"
  [新问题]
  用户: "我叫什么名字？"

AI看到历史记录后: "你叫小明！"
```

**就像一个失忆的人，你每次都把之前的聊天记录给他看一遍，他就"记住"了。**

### 1.3 这个项目是怎么实现的？

整个流程分三步：**存 → 取 → 拼**

```
┌─────────────────────────────────────────────────────┐
│  第1步：存（把聊天记录存到数据库）                      │
│                                                       │
│  用户: "帮我写一段代码"                                 │
│    ↓                                                  │
│  保存到 MySQL 数据库:                                  │
│    消息ID=1, 内容="帮我写一段代码", 类型=USER           │
│                                                       │
│  AI回复: "好的，以下是代码..."                          │
│    ↓                                                  │
│  保存到 MySQL 数据库:                                  │
│    消息ID=2, 内容="好的，以下是代码...", 类型=AI        │
│    关联: reply_id=1 (表示这条是回复消息1的)             │
│                                                       │
├─────────────────────────────────────────────────────┤
│  第2步：取（用户再次发消息时，从数据库取出历史记录）       │
│                                                       │
│  用户: "能加个注释吗？"                                 │
│    ↓                                                  │
│  从数据库取出这个对话的所有消息:                         │
│    消息1: "帮我写一段代码"                              │
│    消息2: "好的，以下是代码..."                         │
│                                                       │
├─────────────────────────────────────────────────────┤
│  第3步：拼（把历史+新消息拼成一个大文本，发给AI）         │
│                                                       │
│  发给AI的内容:                                         │
│    用户: 帮我写一段代码                                 │
│    AI: 好的，以下是代码...                              │
│    用户: 能加个注释吗？  ← 新消息                       │
│                                                       │
│  AI看到完整上下文后回答: "当然可以，注释版代码如下..."     │
└─────────────────────────────────────────────────────┘
```

**就这么简单！AI 的"记忆"就是每次都把聊天记录带着走。**

## 2. AI 是怎么"记住"对话的？

### 2.1 问题：聊天记录太长怎么办？

如果你跟 AI 聊了 100 轮，全部发给 AI，会有两个问题：

1. **太贵了**：AI 按字数收费，每次都带 100 轮历史，费用暴增
2. **装不下**：AI 有字数上限（比如只能看 4000 字），超出的部分看不到了

### 2.2 解决办法：只带最近的几轮对话

就像翻聊天记录，你只看最近的几条，不会从去年的消息开始看：

```
完整历史: 第1轮 → 第2轮 → 第3轮 → ... → 第48轮 → 第49轮 → 第50轮
                                                    ↑        ↑
                                          只取最近10轮发给AI ←┘
                                          (第41~50轮)
```

### 2.3 代码是怎么做的？

```java
// 一个简单的方法，从历史消息中取"最近N轮"
// N = maxContexts（比如设为10）

从最后一条消息开始，倒着往前找:
  跳过没有配对的（没有reply_id的）
  跳过内容为空的
  只取完整的"用户问 → AI答"成对的消息
  取够N对就停

最后反转一下，恢复成时间顺序
```

**这就像一个"滑动窗口"**——窗口大小是 N 轮对话，永远只看最近的内容。

### 2.4 数据库里存了什么？

这个项目用了三张表：

```
📋 对话表 (ai_chat_conversation)
├── 对话标题: "讨论代码优化"
├── 角色设定: "你是一个高级程序员..."
├── 模型参数: 温度0.7, 最大长度2000
└── 上下文大小: 10（最多带10轮历史）

💬 消息表 (ai_chat_message)
├── 消息1: USER   "帮我优化这段代码"           reply_id=空
├── 消息2: AI     "好的，可以从以下几个方面..."  reply_id=1  ← 回复消息1
├── 消息3: USER   "还有别的方法吗？"            reply_id=空
├── 消息4: AI     "当然，还可以..."             reply_id=3  ← 回复消息3
└── ...

📝 日志表 (ai_conversation_log)  — 审计用
├── 记录了每次发给 AI 的完整内容
├── 记录了 AI 的完整回复
├── 记录了用了多少 Token（字数）
└── 记录了是否成功
```

## 3. 为什么 AI 能"逐字输出"？

### 3.1 生活中的例子

想象你在等一个朋友手写一封信，有两种方式：

**方式A：等他写完再给你（普通模式）**
```
你: "帮我写封信"
朋友: （写了5分钟...）"好了，给你"
你: （等了5分钟，一下看到全文）
```

**方式B：他写一个字你就看一个字（流式模式）**
```
你: "帮我写封信"
朋友: "亲"   ← 你马上看到
朋友: "爱"   ← 你马上看到
朋友: "的"   ← 你马上看到
朋友: "小"   ← 你马上看到
朋友: "明"   ← 你马上看到
...
（每个字都实时出现，像打字效果）
```

ChatGPT 那种"一个字一个字往外蹦"的效果，就是方式B。

### 3.2 技术原理（通俗版）

这用到了一个叫 **SSE** 的技术（Server-Sent Events，服务器推送事件）。

普通请求是"一次性"的：
```
客户端：给我结果！
服务端：（处理处理...）给你！
客户端：收到了。
—— 连接关闭
```

SSE 是"持续推送"的：
```
客户端：给我结果，别关连接！
服务端：收..到..我..一..个..字..一..个..字..发..给..你
客户端：亲  爱  的  小  明  （一个一个收到）
—— 发完了才关连接
```

### 3.3 这个项目的实现

```java
// 这是一个简化版的实际代码逻辑

// 1. 声明这是一个"流式"接口
@PostMapping(value = "/send-stream",
             produces = "text/event-stream")  // ← 关键！告诉浏览器这是流式

// 2. 调用 AI 的流式 API
Flux<ChatResponse> stream = chatModel.stream(prompt);
//    ↑
//    Flux 就像一个"水流"，每个字是一个水滴，逐个流出来

// 3. 每个"水滴"（字/词）都立刻发给前端
stream.map(每个字 -> {
    拼到缓冲区;
    立刻返回给前端;  // 用户马上看到这个字
})

// 4. 流结束后的处理
.doOnComplete(() -> {
    // 全部发完了，把完整内容存到数据库
    保存完整的AI回复;
})

// 5. 出错了怎么办
.doOnError(错误 -> {
    // 如果已经生成了部分内容，保存下来（至少不是空白）
    // 如果什么都没生成，删除空的占位记录
})
```

### 3.4 为什么要这样设计？

| 好处 | 解释 |
|------|------|
| **体验好** | 不用等，马上就看到第一个字 |
| **可取消** | 看到一半不满意？直接关掉，不用浪费后面的 Token |
| **容错** | 中断了也不怕，已生成的内容会保存 |

## 4. AI 绘画是怎么做的？

### 4.1 问题：画一张图要好几十秒

AI 画一张图通常需要 10~60 秒。如果让用户一直等，体验很差，还可能超时（HTTP 请求一般有超时限制）。

### 4.2 解决办法：异步模式

这就像一个"取餐叫号"系统：

```
1. 你点餐 → 拿到一个号码牌 → 可以先去做别的事
2. 厨房做菜（异步进行）
3. 你过一会儿拿号码牌来取餐
```

**技术上的实现**：

```
你点"生成图片"
  ↓
后端:
  ① 在数据库创建一条记录: 状态=处理中
  ② 立刻返回给前端: "已收到，你的任务ID是9527"
  ③ 异步线程: 调用 AI 绘画 API（耗时30秒）
      ↓
     成功 → 图片上传到文件服务 → 更新DB: 状态=完成, 图片地址=xxx
     失败 → 更新DB: 状态=失败, 错误信息=xxx

前端:
  ① 拿到任务ID=9527
  ② 每隔2秒轮询: "9527好了吗？"
  ③ 状态变成"完成"后，显示图片
```

### 4.3 为什么写 `getSelf()` 这种奇怪的代码？

```java
// 注意：不能用 this.executeDrawImage()
// 必须用 getSelf().executeDrawImage()

private AiImageServiceImpl getSelf() {
    return SpringUtil.getBean(getClass());  // 从Spring容器拿自己的代理对象
}
```

**为什么？** 因为 `@Async` 注解是通过"代理"来实现异步的。就像：

```
this.doSomething()  →  直接打电话（同步，你得等着）
proxy.doSomething() →  发微信留言（异步，你不用等回复）
```

类内部直接调用 `this.xxx()` 不走代理，所以不会异步。通过 `getSelf()` 拿到代理对象再调用，就能异步了。

通俗理解：**你不能自己给自己发微信，但可以让秘书帮你发。**

### 4.4 Midjourney 的特别处理

Midjourney 更复杂，因为它用的是第三方代理 API：

```
① 提交生成任务 → 拿到 taskId
② 定时扫描（定时任务去查"做好了吗"）
③ Webhook 回调（做好了主动通知你）
④ 下载图片 → 上传到自己的文件服务（防止第三方链接失效）
⑤ 更新数据库
```

## 5. 知识库是怎么让 AI 更聪明的？

### 5.1 问题：AI 不知道你的专属信息

ChatGPT 训练数据是公开的互联网内容。它不知道：
- 你公司的内部文档
- 你的产品说明书
- 你写的技术规范

如果你问它"根据公司规范，这个接口应该怎么设计？"，它只能瞎猜。

### 5.2 解决办法：RAG（检索增强生成）

**通俗理解**：你问 AI 之前，先快速查一下相关文档，把查到的内容和问题一起发给 AI。

```
普通对话:
  你: "公司的请假流程是什么？"
  AI: "我不清楚贵公司的具体规定..."

RAG对话:
  你: "公司的请假流程是什么？"
  ↓
  ① 先把问题转化成向量（一串数字，代表这句话的意思）
  ② 在知识库里搜索相似的向量 → 找到相关段落
  ③ 把找到的段落和你的问题一起发给AI
  ↓
  发给AI的内容:
    "参考以下公司文档:
     <Reference>员工请假需提前1天在OA系统提交申请...</Reference>
     <Reference>紧急请假可事后补交申请...</Reference>

     用户问题: 公司的请假流程是什么？"
  ↓
  AI: "根据公司规定，请假流程如下：1.提前1天在OA提交..."
```

### 5.3 向量是什么？（最简单的解释）

计算机不认识中文，只认识数字。所以要把文字变成数字：

```
"苹果"      → [0.23, 0.67, -0.12, 0.89, ...]  (一串数字)
"香蕉"      → [0.25, 0.65, -0.10, 0.87, ...]  (很接近苹果的数字)
"请假流程"   → [0.81, -0.33, 0.45, -0.21, ...]  (完全不同的数字)
```

意思相近的词，它们的数字也很接近。这样计算机就能判断"哪些文档跟用户的问题相关"。

### 5.4 文档切片：为什么要把文档切碎？

一篇 10000 字的文档，如果整个拿去跟用户问题比较，效果很差：
- 用户问"请假流程"，整篇文档可能只有 100 字相关
- 其他 9900 字是噪音

所以要先**切碎**：

```
原始文档 (10000字)
  ↓ 切片
片段1: "第一条 员工请假需提前1天..."        (200字)
片段2: "第二条 加班调休需填写加班申请..."    (200字)
片段3: "第三章 薪资计算方式如下..."          (200字)
...
  ↓ 向量化
片段1 → [0.81, -0.33, ...]
片段2 → [0.45, 0.12, ...]
片段3 → [-0.23, 0.67, ...]
  ↓ 存入向量数据库

用户问 "请假流程" → 向量检索 → 找到片段1（最相关）
```

### 5.5 自动检测文档类型

这个项目能自动判断你上传的是什么类型的文档，用不同的切片方式：

```java
// 自动检测逻辑（简化版）

如果文件后缀是 .md 并且有很多 "## " 开头的标题:
  → 按标题切（Markdown问答格式，每个标题是一段）
  
如果文件后缀是 .md:
  → 按语义段落切（更智能的分段）
  
其他文件:
  → 按字数切（固定长度分段）
```

## 6. 怎么防止用户发违规内容？

### 6.1 敏感词过滤

原理非常简单：**维护一个敏感词列表，每次用户发消息前检查一遍**。

"你在消息中是否包含列表里的任何一个词？"

```
用户消息: "今天天气真好"
敏感词列表: ["违禁词1", "违禁词2", "广告词"]

检查过程:
  "今天天气真好" 包含 "违禁词1" 吗？→ 否
  "今天天气真好" 包含 "违禁词2" 吗？→ 否
  "今天天气真好" 包含 "广告词"   吗？→ 否
  → 通过 ✓

用户消息: "加我微信 xxxx 免费领取"
检查过程:
  "加我微信 xxxx 免费领取" 包含 "广告词" 吗？→ 是！
  → 拦截 ✗  (抛出异常: SENSITIVE_WORD_DETECTED)
```

### 6.2 怎么让它更快？

每次都去数据库查敏感词列表 → 太慢了。

所以用了**缓存**：

```java
// 第一层：内存缓存（最快，纳秒级）
private volatile List<词> cachedWords;  // volatile = 多线程可见

检查时: 直接从内存读 → 逐个比较 → 完成

// 增删改敏感词时：清空缓存 → 下次使用时重新从数据库加载
添加敏感词时:
  cachedWords = null;  // 清空缓存
  写入数据库;
```

这就像：
- **内存缓存** = 把常用东西放在桌面上，伸手就拿
- **数据库** = 放柜子里，需要走过去拿
- 改动了就清空桌面 → 下次从柜子里重新拿

## 7. 怎么控制用户调用次数？

### 7.1 配额管理

调用 AI API 是花钱的，所以要限制每个用户每天/每月的调用次数。

```
用户配额表:
  用户ID=1001
  每日上限: 100次
  今日已用: 23次
  每月上限: 3000次
  本月已用: 456次

用户发消息时:
  ① 检查: 今日已用(23) < 每日上限(100)？→ 是，通过
  ② 已用次数+1 → 24次
  ③ 继续处理消息...

如果今日已用 ≥ 每日上限:
  → 返回错误: "今日调用次数已用完"
```

**重新生成时为什么不扣配额？**

因为"重新生成"不是新请求——用户对同一个问题不满意，要求换个回答。这不应该再收费。所以代码里有一个 `skipQuotaCheck` 参数：

```java
// 正常发送: 检查配额
sendMessage(req, userId, skipQuotaCheck=false)

// 重新生成: 跳过配额检查
sendMessage(req, userId, skipQuotaCheck=true)
```

## 8. 怎么同时支持十几个不同的 AI 模型？

### 8.1 问题：每个 AI 平台的接口都不一样

```
OpenAI:     用 /v1/chat/completions 接口，参数叫 "model"
Claude:     用 /v1/messages 接口，参数叫 "model"  
通义千问:    用 /v1/chat/completions 接口，参数叫 "model"
文心一言:    用 /v2/chat/completions 接口，参数叫 "model"
（实际上它们大同小异，但细节不同）
```

如果每种都写一套代码，维护起来会疯掉。

### 8.2 解决办法：工厂模式 + 策略模式

**通俗理解**：就像万能遥控器。

```
万能遥控器（AiModelFactory）:
  ├── 按"创维"按钮 → 创维电视模式（OpenAI）
  ├── 按"海信"按钮 → 海信电视模式（Claude）  
  ├── 按"小米"按钮 → 小米电视模式（通义千问）
  └── ...

你用遥控器按一个按钮，它自动切换对应模式。
你不知道也不关心每种电视的红外编码是什么。
```

代码上的实现：

```java
// 你用的接口（统一入口）
模型服务.getChatModel(模型ID)  → 返回一个 ChatModel

// 内部怎么做的（你不需要关心）
switch(平台) {
    case OPENAI:    → 创建 OpenAI 的 ChatModel
    case DEEPSEEK:  → 创建 DeepSeek 的 ChatModel（豆包/混元/星火也复用这个）
    case ANTHROPIC: → 创建 Claude 的 ChatModel
    case OLLAMA:    → 创建本地 Ollama 的 ChatModel
    ...
}
```

### 8.3 密钥和模型分离

```
ai_api_key 表            ai_model 表
┌──────────────┐        ┌──────────────────┐
│ 名称: OpenAI密钥│       │ 名称: GPT-4o      │
│ 密钥: sk-xxx   │←─────│ key_id: 指向上面   │
│ URL: api.open..│       │ temperature: 0.7  │
└──────────────┘        │ maxTokens: 2000   │
                         │ maxContexts: 10   │
                         └──────────────────┘

好处: 换密钥时只改 ai_api_key，所有用这个密钥的模型自动更新
```

## 9. AI 写作/思维导图/音乐是怎么做的？

### 9.1 它们的共同模式

AI 写作、AI 思维导图、AI 音乐——虽然功能不同，但实现模式完全一样：

```
① 设定"角色" → 让 AI 扮演特定身份
② 构造"问题" → 把用户参数拼成一段话
③ 发送给 AI → 拿到结果
④ 保存结果 → 更新数据库
```

### 9.2 AI 写作的例子

```
用户选择:
  类型: 写作
  主题: "AI技术发展趋势"
  格式: 博客文章
  语气: 专业
  语言: 中文
  长度: 中等

系统构造的 Prompt:
  [系统消息]
  "你是一个专业的科技写作助手，擅长撰写有深度的科技文章。"

  [用户消息]
  "请帮我撰写一篇关于 AI技术发展趋势 的 博客文章，
   语气要求 专业，语言使用 中文，字数要求 中等"

  ↓ 发送给 AI
  ↓ 流式输出 "随着人工智能技术的飞速发展..."
```

**参数从哪里来？** 从字典表翻译：

```
用户选: format=1 → 字典翻译 → "博客文章"
用户选: tone=2   → 字典翻译 → "专业"
用户选: length=3 → 字典翻译 → "中等"
```

### 9.3 AI 音乐的例子

AI 音乐用的是 Suno API（第三方服务），和 AI 绘画类似也是**异步模式**：

```
① 创建2条数据库记录（Suno一次生成2首）
② 返回ID给前端: [id1, id2]
③ 异步线程:
    描述模式: SunoApi.generate("一首欢快的流行歌曲")
    歌词模式: SunoApi.customGenerate("自定义歌词...")
④ 生成完成后: 下载音频文件 → 上传到自有文件服务 → 更新DB

⑤ 定时任务 syncMusic():
    扫描所有"进行中"的任务
    批量查询 Suno → 拿到进度 → 更新DB
```

## 10. 总结：记住这几个核心思路就够了

### 🎯 核心思路一：AI 没有记忆，记忆是"拼"出来的

```
每次发消息 = 历史记录 + 新消息 → 一起发给AI
控制记忆长度 = maxContexts（只带最近N轮）
```

### 🎯 核心思路二：慢任务用异步，快任务用同步

```
聊天（快）→ 同步，等结果返回
绘画/音乐（慢）→ 异步，先返回ID，后台慢慢做，前端轮询
```

### 🎯 核心思路三：相似的功能用统一模板

```
AI写作/AI思维导图/AI辅导 = 
  ① 设定角色(SystemMessage)
  ② 构造问题(UserMessage)  
  ③ 调用AI
  ④ 保存结果

不同的只是：
  - 角色描述（写作助手 vs 思维导图专家 vs 老师）
  - 问题模板（不同的 Prompt 格式）
```

### 🎯 核心思路四：缓存让一切更快

```
敏感词 → 内存缓存（纳秒级）
模型实例 → 对象池缓存（避免重复创建连接）
统计日志 → 异步写入（不阻塞主流程）
```

### 🎯 核心思路五：工厂模式让系统可扩展

```
新增一个 AI 平台？
  → 加一个枚举值
  → 实现/复用 ChatModel
  → 在工厂里注册

新增一个 AI 功能？
  → 加 Controller（接收请求）
  → 加 Service（构造 Prompt + 调 AI）
  → 加 DO/Mapper（保存结果到数据库）
```

### 📊 各功能的实现方式对照表

| 功能 | 调用方式 | 返回方式 | 耗时 | 核心挑战 |
|------|---------|---------|------|---------|
| AI 对话 | 同步/异步 | 一次性/SSE流式 | 1~10秒 | 上下文管理 |
| AI 绘画 | 异步 | 先返回ID，再轮询 | 10~60秒 | 超时处理 |
| AI 音乐 | 异步 | 先返回ID，再轮询 | 30~120秒 | 批量同步 |
| AI 写作 | 同步流式 | SSE逐字输出 | 3~15秒 | Prompt模板 |
| 知识库RAG | 同步 | 嵌入对话流程 | 毫秒级 | 向量检索精度 |
| 敏感词 | 同步 | 即时判断 | 微秒级 | 缓存一致性 |
